## 一、产品定位与命名

### 1.1 产品名称

**OfferMind**

> **释义**：Offer（面试机会/录用通知）+ Mind（心智/决策）。寓意求职者运用“职业心智”，主动做出清醒的职业选择，而非被动等待机会。

### 1.2 产品定位

**AI 职业顾问。**

这是一个**“决策支持型”**产品，而非“工具型”产品。区别在于：


|          | 工具（Tool）               | 顾问（Advisor）              |
| :------- | :------------------------- | :--------------------------- |
| **角色** | 被动响应，你问什么它答什么 | 主动引导，告诉你应该考虑什么 |
| **输出** | 一份报告                   | 一条可执行的路径             |
| **关系** | 一次性使用                 | 长期陪伴                     |
| **价值** | 提高效率                   | 降低决策风险                 |

### 1.3 核心价值主张

> **“让每一次投递都是清醒的选择，让每一段经历都指向更好的未来。”**

### 1.4 一句话描述

> AI 职业顾问。帮你诊断简历、研究岗位、规划路径——在投递之前，先看清全貌。

### 1.5 品牌 Slogan

> **Know Before You Apply.**
> （投递之前，先看清。）

## 二、产品愿景与使命

### 2.1 愿景

**成为每个职场人的“职业决策第一站”。**

在投递简历之前、在跳槽之前、在转型之前，先来 OfferMind 做一次完整的职业诊断与规划。

### 2.2 使命

**用 AI 消除职业决策中的信息不对称。**

求职者不应该因为“不知道这个部门在做什么”而错过好机会，也不应该因为“不了解岗位的真实要求”而浪费时间和信心。我们帮求职者看清全貌，做出属于自己的判断。

### 2.3 产品原则

1. **尊重事实边界**：AI 只在用户已有经历基础上优化表述，不新增虚构内容
2. **信息透明**：用户知道 AI 从哪里获取信息、如何得出结论
3. **自主决策权**：AI 提供判断和推荐，最终决定权永远在用户手中
4. **长期主义**：不鼓励“速成”，鼓励“可持续的职业成长”

### 2.4 伦理红线

1. 不呈现个人联系方式（电话、邮箱、社交账号）
2. 不生成“套近乎”话术（如“您可以跟面试官聊他的论文来拉近关系”）
3. 所有结论附带可追溯来源
4. 不主动推断未公开信息
5. 搜索结果为空时如实告知，不编造
6. 职业规划不制造焦虑，不推销不切实际的目标

| 用户类型       | 描述                                 | 核心场景                           | 使用频次              |
| :------------- | :----------------------------------- | :--------------------------------- | :-------------------- |

| **长期成长者** | 已有明确方向、希望持续成长的专业人士 | 追踪目标岗位要求、规划学习路径     | 月均 2-3 次（持续）   |

### 4.2 模块一：简历多维诊断

**功能定位**：用户进入 OfferMind 的第一站。用六级维度标准对简历进行全面体检。

**输入**：

- 简历：PDF / Word / 图片（OCR 自动识别）
- JD：文本粘贴 / 图片上传（OCR 自动识别）

**输出**：结构化诊断报告，包含：


| 报告板块           | 内容                                                | 用户价值                 |
| :----------------- | :-------------------------------------------------- | :----------------------- |
| **总分与等级**     | 0-100 分 + 优秀/良好/合格/待改进                    | 一目了然知道简历水平     |
| **六级维度雷达图** | 内容质量/结构格式/岗位匹配/诊断优化/公平性/模型能力 | 看清强弱项分布           |
| **JD 拆解展示**    | 必须项/加分项/痛点项 三列展示                       | 知道 AI 如何理解这个岗位 |
| **分项评分与评语** | 每个二级指标打分+具体分析                           | 知道“为什么是这个分数” |
| **风险预警清单**   | 时间悖论/职级不匹配/表述夸大等                      | 提前排雷，避免面试穿帮   |
| **优化建议列表**   | 动词升级/数字补全/关键词嵌入/表述精炼               | 可直接执行的修改方向     |
| **改写示范段落**   | 原文 vs 优化版对比                                  | 看到“好的表述长什么样” |

**关键约束**：

- 硬性条件不满足 → 总分上限 59 分，标记“不建议投递”，说明原因
- 所有建议必须附带具体示范，不出“建议补充数据”这种空泛表述
- 如简历全文无数字，高亮提醒“建议补充量化成果”

### 4.3 模块二：定制化简历生成

**功能定位**：一份原始简历 → 针对不同 JD → 自动生成多个定制版本。

**用户场景**：小明同时投递 A、B、C 三家公司的产品经理岗位，JD 侧重点各不相同。A 公司强调“用户增长”，B 公司强调“数据分析”，C 公司强调“跨部门协作”。他不需要手动改三份简历，系统自动生成三个版本。

**核心逻辑**：

```
用户原始简历（母版 · 永不覆盖）
        │
        ├── + JD_A（强调微服务） → 版本 A：突出微服务经历，增加技术细节
        ├── + JD_B（强调高并发） → 版本 B：突出性能优化经历，量化结果
        └── + JD_C（强调团队管理）→ 版本 C：突出带团队经历，弱化纯技术细节
```

**AI 自动执行的五步操作**：


| 步骤          | 操作内容                                                 | 约束               |
| :------------ | :------------------------------------------------------- | :----------------- |
| 1. 经历排序   | 根据 JD 侧重点，将最相关的经历置顶，弱相关的折叠或精简   | 不删除任何经历     |
| 2. 关键词注入 | 在相关经历中自然嵌入 JD 核心关键词（含同义词）           | 不超出原文事实范围 |
| 3. 表述加权   | 高相关经历延长描述篇幅、增加技术/业务细节                | 不虚构新增内容     |
| 4. 动词升级   | 根据岗位类型（技术/产品/设计/市场/管理）匹配对应的强动词 | 与原文事实一致     |
| 5. 弱项处理   | 对 JD 要求但简历缺失的技能，如实提示“建议补充”，不编造 | 保持事实边界       |

**岗位类型风格映射表**：


| 岗位类型  | 侧重方向                     | 表述风格                                   | 动词偏好                     |
| :-------- | :--------------------------- | :----------------------------------------- | :--------------------------- |
| 技术岗    | 技术深度、架构能力、性能指标 | 强调技术选型、系统设计、代码质量、量化结果 | 搭建、重构、优化、设计、实现 |
| 产品岗    | 用户洞察、需求分析、业务闭环 | 强调用户调研、需求转化、数据验证、商业价值 | 洞察、定义、推动、验证、迭代 |
| 设计岗    | 设计思维、用户研究、视觉表达 | 强调设计方法论、用户反馈、作品集关联       | 构思、设计、测试、优化、落地 |
| 市场/运营 | 增长策略、转化漏斗、ROI      | 强调渠道运营、转化数据、活动策划           | 策划、运营、增长、转化、撬动 |
| 管理岗    | 团队领导、战略决策、资源整合 | 强调团队规模、项目管理、跨部门协同         | 带领、制定、协调、决策、推动 |

### 4.4 模块三：目标部门深度研究

**功能定位**：帮助求职者在投递前，了解目标岗位背后的部门、人物、研究方向——消除信息不对称。

**触发方式**：诊断报告生成后，用户点击“🔍 研究目标部门”按钮（也可在首页独立触发）。

**输入**：从 JD 中自动提取的公司名 + 岗位名 + 部门名（用户可手动修正）。

**输出**：《目标部门深度研究报告》


| 报告板块             | 内容                                     | 信息来源                   | 呈现方式        |
| :------------------- | :--------------------------------------- | :------------------------- | :-------------- |
| **部门定位**         | 规模、核心职能、组织架构位置             | 官网/年报/招聘页           | 摘要卡片        |
| **关键人物画像**     | 负责人姓名、教育背景、职业轨迹、研究兴趣 | LinkedIn/学术主页/公司官网 | 人物卡片        |
| **研究方向图谱**     | 近期论文/专利标题、技术博客关键词        | Google Scholar/专利库/博客 | 关键词云+时间线 |
| **部门动态**         | 近期大事件、战略调整、新项目             | 新闻/公司博客              | 动态列表        |
| **校友网络（匿名）** | 有相同教育背景的成员数量统计             | LinkedIn 交叉匹配          | 匿名化统计卡片  |

**五条伦理红线（技术+产品双重约束）**：

1. **不呈现个人联系方式**（电话、邮箱、社交账号）——系统搜索时自动过滤
2. **不生成“套近乎”话术**（如“您可以跟面试官聊他的论文”）——系统指令明确禁止
3. **所有结论附带可追溯来源**——每条信息附 URL 链接
4. **不主动推断未公开内容**——搜索结果为空时如实告知，不编造
5. **搜索信息少时诚实说明**——“该部门的公开信息较少，以下结果仅供参考”

### 4.5 模块四：职业道路规划

**功能定位**：帮助用户在更长的时间尺度上规划自己的职业发展。这是 OfferMind 从“单次工具”走向“长期顾问”的关键模块。

**触发方式**：

- 完成一次诊断后，点击“📈 规划我的职业路径”
- 或在首页直接进入“职业规划”入口（不需上传 JD，只需上传简历+输入目标方向）

**输入**：

- 用户当前简历（必填）
- 目标方向：具体岗位名称 / 上传 JD / 自由输入“我想做 XXX”（三选一）
- 职业偏好问卷（选填）：技术深度 vs 管理路线 vs 业务导向

**输出**：《职业发展路径规划报告》


| 报告板块                     | 内容                                              | 数据来源                 |
| :--------------------------- | :------------------------------------------------ | :----------------------- |
| **Part 1：当前职业画像**     | 硬技能清单、软技能评估、经验领域、职级定位        | 简历解析 + 能力模型库    |
| **Part 2：目标岗位能力模型** | 目标岗位需要的硬技能、软技能、经验、资质          | 行业能力模型库 + JD 分析 |
| **Part 3：能力差距分析**     | 硬技能差距/软技能差距/经验缺口/资质缺失，分优先级 | Part 1 vs Part 2         |
| **Part 4：学习路径推荐**     | 分阶段推荐课程/书籍/项目/证书，附带具体链接       | 外部课程库/认证库        |
| **Part 5：发展时间线**       | 3-6个月/1年/3年 里程碑                            | 基于差距分析推导         |
| **Part 6：部门联动（可选）** | 与模块二联动：目标部门研究方向映射到学习建议      | 模块二输出               |

**Part 4：学习路径推荐的呈现结构**：

```
📚 你的个性化学习路径

【🔴 优先级：高】立即开始（1-3个月内）
  ├── 课程：《SQL 基础入门》（Coursera）
  ├── 书籍：《精益数据分析》（豆瓣链接）
  └── 项目：用公开数据集完成一次用户增长分析
  → 预计投入：每周3-4小时

【🟡 优先级：中】近期规划（3-6个月内）
  ├── 证书：Google Analytics 认证
  └── 实践：在目前工作中主动承担数据分析职责
  → 预计投入：每周2-3小时

【🟢 优先级：低】长期积累（6-12个月以上）
  ├── 深入学习：Python 数据分析库（Pandas/NumPy）
  └── 拓展：关注行业增长案例，建立自己的方法论库
  → 预计投入：每周1-2小时
```

**Part 5：发展时间线的呈现形式**：

### 4.6 模块五：产品自评估与持续迭代

**功能定位**：OfferMind 内置的“体检系统”——持续监测产品自身的诊断质量、公平性和稳定性。

**三层漏斗评估体系**：

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│ L1：自动化客观评测层（40% 权重）                                                   │
│ 自动计算：数字密度、关键词覆盖率、格式合规、实体抽取 F1值                           │
├─────────────────────────────────────────────────────────────────────────────────────┤
│ L2：LLM-as-Judge 评测层（40% 权重）                                                │
│ 用 Claude 3.7 / GPT-4o 作为裁判，按六级维度评分，每个样本跑 3 次取中位数            │
├─────────────────────────────────────────────────────────────────────────────────────┤
│ L3：人工专家抽检层（20% 权重）                                                     │
│ HR 专家盲审 10% 样本，计算 ICC（组内相关系数），校准机器评分偏差                    │
└─────────────────────────────────────────────────────────────────────────────────────┘
```

**评测数据集**：


| 评测维度 | 选用数据集              | 规模               | 用途              |
| :------- | :---------------------- | :----------------- | :---------------- |
| 内容质量 | CareerCorpus            | 302 份（专家标注） | Ground Truth 参照 |
| 结构格式 | ResumeBench             | 2,500 份           | 结构化理解评测    |
| 岗位匹配 | RJDB / PJB              | 5万+ 组            | 匹配推理能力      |
| 诊断优化 | Resume Training Dataset | 2.2万 份对话       | 诊断建议质量      |
| 公平性   | FAIRE + FairCV          | 多组               | 偏见检测          |
| 模型能力 | ResumeBench + JobResQA  | 多语言             | 跨语言/抽取能力   |

**可信度校验机制**：


| 校验项                      | 合格阈值  | 不达标处理                       |
| :-------------------------- | :-------- | :------------------------------- |
| 人机一致性（ICC）           | > 0.75    | 调整裁判模型提示词               |
| 裁判模型自洽性（3次标准差） | < 3 分    | 细化评分锚点                     |
| 维度区分度（各维度方差）    | > 10      | 拉大量纲                         |
| 公平性校验（FAIRE）         | 波动 < 2% | 增加“禁止考虑人口统计属性”指令 |

**持续迭代闭环**：

```
用户使用 → 收集 Bad Case → 每周人工审查 → 更新评测集 →
→ 重新跑 L1/L2/L3 → 定位退化维度 → 优化模型/提示词 →
→ 回归测试 → 部署新版本 → 继续用户使用
```
